# **0\. このドキュメントについて**

下腦（しもの）という架空世界の気候・植生・歴史を、物理的に整合したシミュレーションで決定するための手順書です。

GSM（全球スペクトルモデル）と NeuralGCM を参考に、地球での検証を経て下腦の条件に拡張します。

ナレッジはPodcast・動画・Wikiと連動し、試行錯誤の過程そのものをコンテンツ化することを目標とします。

# **2\. ②↔③ 地形↔気候 収束ループ**

## **2-1. ループの目的**

気象は地形の総体として現れます。降水量・気温・植生すべてが山脈・海岸線・高度に依存するため、地形と気候を同時に最適化する必要があります。

## **2-2. 最初の仮地形**

* まず「最低限の仮地形球」を仮定してシミュレーションを開始

* 降水量の必要条件を満たすために山をここに置く

* 気温の必要条件を満たすためにここの高度を上げる

* この「ファジーな地理の考え方」を NeuralGCM の ML モジュールが担う

## **2-3. 収束アルゴリズム**

目的関数:

  minimize  Σᵢ |気候指標ᵢ(地形) − 目標値ᵢ|

  subject to: 地形の物理的整合性、液体水の存在、①の歴史条件

*※ JAX の自動微分で ∂気候/∂地形 の勾配を計算し、どこの高度を上げると降水量がどれだけ変わるかを数値的に求めながら地形を更新します。*

## **2-4. 停止条件**

* 全①条件（歴史・国規模から定義した気候レンジ）を満たす

* 液体水が存在する（惑星が凍結も蒸発もしていない）

* 赤道〜極温度差が設定レンジ内

* 降水がある程度の範囲に分布している

* 大気が数値的に安定（発散しない）

# **3\. GPV 初期値の最尤推定（MLE）補完**

## **3-1. 問題**

地図（陸海マスク）から気温・気圧・湿度・風速など GPV の数十変数が得られない。「もっともらしい初期値」を自動生成する必要があります。

## **3-2. アプローチ**

* ERA5 の 40年統計分布を事前分布として使用

*   p(T, P, q, u, v | lat, lon, month, land\_mask)

* 下腦固有の拘束（表面温度・大気圧）を尤度として掛け合わせ

* MAP 推定（Maximum A Posteriori）で初期値を生成

*※ 完全に 0 から乱数で始めるよりスピンアップが大幅に短縮されます。*

## **3-3. 初期値と力学的整合性（Initialization Shock）**

* 手置き初期値が地衡風バランスを満たさないと計算開始直後に架空の重力波が発生して発散

* NeuralGCM の encode() がこれを吸収する可能性があるが下腦では個別検証が必要

* σ座標（地形追従座標）と ERA5 圧力座標の変換が encode()/decode() の核心

# **4\. 放射スペクトル変更と物理的整合性**

## **4-1. 「温かさ」の再定義**

恒星スペクトルが異なると地表での短波放射量・波長分布が変わり、「気温 280 K \= 水が液体」という地球の経験則が成立しない可能性があります。

## **4-2. 手順**

* 放射平衡温度を先行計算

*   T\_eq \= ( L★ × (1 − アルベド) / (16π σ a²) )^0.25

* T\_eq を基準に初期温度場を組み、相転移温度域を確認してから GPV を設定

## **4-3. 氷↔水 相転移の検証**

* アルベドフィードバックの正常動作確認（氷が増えると反射率が上がりさらに冷えるループ）

* 液体水が存在する温度・気圧の組み合わせを個別シミュレーションで確認

# **5\. 技術的課題と対処方針**

| 課題 | 内容 | 対処方針 |
| :---- | :---- | :---- |
| **GPV 初期値 MLE 補完** | 地図（陸海マスク）から気温・気圧・湿度・風速など数十変数を埋める | ERA5 の統計分布を事前分布として MAP 推定。地球で検証してから拡張 |
| **初期値と天気言語の整合** | 手置き初期値が地衡風バランスを満たさないと計算開始直後に発散（initialization shock） | NeuralGCM の encode() で吸収。下腦では個別検証が必要 |
| **放射スペクトル変更** | 恒星スペクトルが異なると「温かさ」の定義が変わり相転移温度域がずれる | 放射平衡温度を先行計算してから初期温度場を設定。氷↔水相転移を個別検証 |
| **②↔③ 収束ループ** | 降水条件のために山を置く → 気候が変わる → 地形再調整、の循環 | JAX 自動微分で ∂気候/∂地形 の勾配を取り、目的関数を最小化 |
| **順序の整合性** | 気象はプレート・植生・海流・高度の総体。順序を誤ると不整合 | ②↔③ を反復させ、④で ① との照合を毎回行う。段階的検証 |
| **計算資源** | 毎回フルシミュレーションは不可能 | NeuralGCM の ML モジュールが「ファジー地理」の暗黙知を代替 |

# **6\. 下腦の惑星パラメータ設計空間**

以下のパラメータは議論・Podcast で決定し逐次更新します。

| パラメータ | 地球の値 | 下腦での選択肢 | 気候への影響 |
| :---- | :---- | :---- | :---- |
| **自転周期** | 24 h | 任意 | コリオリ力 f の強さ → ハドレー循環幅 |
| **軸傾斜** | 23.5° | 任意 | 季節性の強弱 |
| **恒星スペクトル** | G型（太陽） | K/F型など | 放射平衡温度・氷アルベドフィードバック |
| **大気組成** | N₂/O₂/CO₂ | 任意 | 温室効果・放射窓 |
| **大陸配置** | 既知 | 設計する | 海陸分布 → モンスーン・砂漠形成 |

*※ コリオリ力を変えると大気大循環セルの数と位置が変わります。自転が遅い惑星では全球規模のハドレー循環になり、熱帯が大陸全体に広がる可能性があります。*

# **7\. SeesaaWiki 編集履歴の自動収集**

## **7-1. 目的**

⑥「Wiki更新順の球体プロット」のデータ源として、全記事の編集履歴を時系列で記録します。

## **7-2. URL 構造**

* ページ一覧:  https://seesaawiki.jp/WIKI\_NAME/l/

* 各ページ:   https://seesaawiki.jp/WIKI\_NAME/d/PAGE\_NAME

* 履歴ページ: https://seesaawiki.jp/WIKI\_NAME/history/PAGE\_NAME

* 差分ページ: https://seesaawiki.jp/WIKI\_NAME/diff/PAGE\_NAME/REVISION\_ID

## **7-3. 文字コード**

SeesaaWiki は EUC-JP ですが Shift-JIS の機種依存文字（①など）が混入することがあります。

* Python のデコードには euc\_jis\_2004 を使用

## **7-4. スクレイパーの実行**

python seesaa\_history\_scraper.py \--wiki YOUR\_WIKI\_NAME \--output shimono\_changelog

出力: shimono\_changelog.json（全履歴）および shimono\_changelog.csv（スプレッドシート用）

## **7-5. 重大変更の判定基準**

* 追加行数 ≥ 5 行

* 削除行数 ≥ 3 行

* 編集コメントに「新設・作成・追加・改定・建国・戦争・条約」等のキーワードを含む

*※ 閾値と重大キーワードは seesaa\_history\_scraper.py の MAJOR\_THRESHOLDS / MAJOR\_KEYWORDS で調整できます。*

# **9\. NeuralGCM のハイブリッド構造と役割**

## **9-1. アーキテクチャ**

* 物理ダイナミクスコア: Navier–Stokes 方程式を数値ソルバーで解き大規模大気循環を計算

* ML モジュール（Haiku/JAX）: 雲生成・降水など未解像過程をニューラルネットでパラメタライズ

* encode(): 圧力座標 → σ座標（力学的整合性を保証）

* decode(): σ座標 → 圧力座標（ERA5 比較用）

## **9-2. 下腦での活用**

* ML モジュールが「ファジー地理」の暗黙知を担い、毎回フルシミュレーションしなくてよい

* JAX 自動微分で ∂気候/∂地形 の勾配を取り、地形最適化ループに組み込む

* 地球で学習済みの重みを転移学習のベースとして使い、下腦に微調整
